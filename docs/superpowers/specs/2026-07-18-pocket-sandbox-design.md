# Sandbox-примерка «что если» на кармашке (ANO-16)

**Дата:** 2026-07-18. **Linear:** ANO-16 (родитель ANO-8). Дизайн утверждён Кириллом 2026-07-18 (все развилки — в комменте ANO-16). Киллер-фича фазы «Выход».

## 1. Суть

Примерить хотелку в прогноз кармашка БЕЗ коммита в план: включил тумблер — увидел две линии («как сейчас» / «с примеркой») и два числа; пощёлкал, покрутил, ничего не сохранилось. Скоуп-переключатель кармашка (до дохода / +N мес / до реализации хотелки) — вторая половина фичи: одна примерка на разных скоупах отвечает на разные вопросы. Отдельная кнопка «зафиксировать» переводит решение в реальный план.

Стори (коммент ANO-16 от 2026-07-02): «кармашек 50к; хочу отпуск через полгода — 20к/мес; добавляю хотелку в анализ и вижу, как меняется кармашек на каждом скоупе».

## 2. Решения (утверждены 2026-07-18)

1. **Скоуп v1 = вариант Б:** примерка существующих хотелок/копилок (вкл/выкл + правка суммы/даты/растяжки) + ad-hoc трата «сумма + дата». Правка существующего плана и «доход −30%» — вне v1. Исключение: FIXED-элементы можно ВЫКЛЮЧАТЬ (примерка отказа) — это жонглирование хотелками, не правка плана.
2. **Оси:** основной экран — кармашек (дневная траектория по скоупу). Капитал/обязательства — по кнопке «что с капиталом» (реюз месячного расчёта wishlist as-is, подписан как приблизительный).
3. **UI:** страница /wishlist пересобирается в окно примерки. Старый 36-месячный график уходит из главной роли (остаётся внутри капитал-кнопки).
4. **Тумблер ≠ статус:** тумблер — эфемерная примерка (Dashboard не трогает); «зафиксировать» → FIXED → входит в реальный кармашек. OPEN и FIXED рисуются по-разному.
5. **Архитектура:** считает сервер тем же PocketEngine с подменённым входом; контракт сразу готов к гибриду (сервер отдаёт дневные дельта-векторы; браузер в будущем сможет складывать линию сам, но НИКОГДА не повторяет правила движка).
6. **Ползунок растяжки у любой хотелки:** разовую можно «копить по X/мес» — математика копилки.
7. **FIXED-копилки начинают резервироваться в реальном кармашке** (см. §6) — единственное изменение семантики ядра.

## 3. Архитектура (бэк)

```
PocketInputAssembler (выделен из PocketService: выборки + резолюция горизонта)
        ├── PocketService.getPocket()          → GET /pocket        (как раньше)
        └── PocketSandboxService.simulate()    → POST /pocket/sandbox
                 baselineInput = assembler.build(scope)
                 fittedInput   = baselineInput − excluded + развёрнутые tryOn-события
                 → PocketEngine.calculate(baselineInput), calculate(fittedInput)
```

- `PocketEngine` не меняется примеркой вообще (примерка = подмена входа). Единственная правка движка — новая breakdown-строка взносов (§6).
- Раскладка «item → дневные события» — чистый статический хелпер (`SandboxLayout` или аналог), покрыт табличными тестами; тот же хелпер используется и для резервирования FIXED-копилок в реальном кармашке (§6) — одна раскладка, два потребителя.
- `WishlistSimulationService` (месячная математика) не трогается: питает капитал-кнопку. `GET /wishlist/simulation` и `POST /wishlist/simulation/recompute` остаются. Наложение FIXED на /strategy (`computeDeltaForFixedItems`) не трогается.
- Программируемость (задел ANO-27): endpoint самодостаточен — произвольный набор параметров без UI-состояния; ответ машиночитаем (pocket/minPoint обеих веток). Batch-режим сознательно не делаем.

## 4. Контракт POST /api/v1/pocket/sandbox

Запрос:

```json
{
  "scope": "MONTHS:3",                  // тот же парсер PocketScope, что GET /pocket
  "tryOn": [
    { "ref": {"type": "EVENT", "id": "…"},   // существующая хотелка; null для ad-hoc
      "amount": 80000,
      "date": "2026-12-14",                  // обязательна
      "stretchMonths": 5,                    // 0/absent = разовая трата в дату
      "creditRate": null, "creditTermMonths": null },   // только для kind CREDIT
    { "ref": null, "amount": 12000, "date": "2026-08-20" }          // ad-hoc
  ],
  "exclude": [ {"type": "FUND", "id": "…"} ]   // FIXED-элементы, выключенные из baseline
}
```

Ответ:

```json
{
  "baseline": PocketResultDto,          // реальный кармашек (с учётом §6)
  "fitted":   PocketResultDto,          // с примеркой
  "itemDeltas": [                       // задел под гибрид
    { "ref": …, "days": [ {"date": "2026-08-15", "delta": -16000}, … ] }
  ],
  "items": [ …полный список хотелок и копилок (OPEN + FIXED, события + фонды),
             с дефолтными параметрами примерки… ]
}
```

- Первый вызов с пустыми `tryOn`/`exclude` инициализирует окно: baseline = fitted, `items` — что показывать в списке. Отдельный listing-endpoint не нужен.
- `itemDeltas` содержит вектор и для каждого excluded-элемента (положительный — «деньги вернулись»).
- **Инвариант (контрактный тест):** `baseline.trajectory[d] + Σ itemDeltas[*].days[d] = fitted.trajectory[d]` для каждого дня d. Страхует будущий гибрид: фронту разрешено только складывать готовые векторы.
- Скоуп «до реализации» = существующий `DATE:<targetDate включённой хотелки>` — фронт подставляет сам, бэку ничего не нужно.

## 5. Раскладка по дням

- **Разовая** (stretchMonths 0): один расход `amount` в `date`.
- **Растянутая/копилка** (stretchMonths n ≥ 1): n равных взносов `amount / n` (HALF_UP; последний взнос добирает остаток копеек, чтобы сумма сходилась точно), начиная со следующего календарного месяца после asOfDate, последний — в месяц цели.
  - **День взноса в месяце = дата первого планового дохода месяца** (PLAN PLANNED INCOME из уже выбранных событий траектории, хотелки исключены): откладывают с зарплаты. Точка траектории — конец дня (движок складывает день нетто), поэтому взнос дня зарплаты не продавливает минимумы ДО зарплаты — ради этого и дневная гранулярность.
  - Нет планового дохода в месяце → 1-е число (консервативно).
  - В месяц цели взнос не позже `date`; если первый доход месяца позже — взнос в сам день `date`.
  - Диапазон: `1 ≤ stretchMonths ≤ месяцев до цели` (следующий месяц…месяц цели включительно); вне диапазона → 400.
  - Сама покупка счёт не трогает (деньги уже ушли взносами — покупают из накопленного).
- **Кредит** (creditRate + creditTermMonths заданы): выдача и покупка взаимно гасятся (нетто 0 для счёта); расход = аннуитетный PMT ежемесячно, начиная с месяца, следующего за месяцем `date`, в день числа покупки (кламп по длине месяца — правило day-of-month как в recurring). PMT — та же формула, что `WishlistSimulationService.computeCreditDelta` (вынести в общий хелпер, не дублировать). Хвост серии за горизонтом просто не попадает в траекторию.
- Внутри примерки все синтетические события — EXPENSE с описанием item'а (`minPoint.drivenBy` и tooltip показывают виновника по имени).

## 6. Изменение ядра: резервирование FIXED-копилок

Сейчас непоследовательно: FIXED-событие с датой в реальной траектории есть (`allowedInTrajectory`), FIXED-копилка — нет («взносы не резервируются», известная дыра ANO-12). Без этого тумблер FIXED-копилки в примерке бессмыслен (выключать нечего).

- `PocketInputAssembler` разворачивает **датированные неконвертированные FIXED-копилки** в синтетические взносы-EventSnapshot той же раскладкой §5: месячный взнос = `(targetAmount − currentBalance) / месяцев до цели` (правило существующего планировщика фондов). Без даты (Доска, Машина) — не резервируются.
- `EventSnapshot` получает признак синтетического взноса (например, `syntheticKind = SAVINGS_CONTRIBUTION`); движок считает их обычными расходами, но breakdown выделяет **новую строку `SAVINGS_CONTRIBUTIONS` («Взносы в копилки, N шт»)** с details по именам копилок. Инвариант breakdown расширяется этой строкой (тесты движка обновляются).
- **Осознанное следствие (решение Кирилла):** число на Dashboard честно уедет вниз. На данных Кирилла: Египет (150 000 к 30.09) ≈ −75 000/мес авг–сен + Горнолыжка (80 000 к 14.12) ≈ −16 000/мес. Лекарство, если копить не собирался — перевести копилку в OPEN.
- Границы: скоуп «до дохода» взносов будущих месяцев не видит — этим резервирование мягче, чем кажется; конвертированные и DISMISSED копилки не участвуют.

## 7. Экран (пересобранный /wishlist)

- **Шапка:** число кармашка (то же, что Dashboard) + скоуп-кнопки (до дохода / 2-й доход / 3 мес / 6 мес; «до реализации» появляется при включённой датированной хотелке) + число «с примеркой» и разница.
- **График:** две дневные линии от сегодня до конца скоупа — baseline приглушённая, fitted яркая; минимумы подписаны на обеих. Реюз `PocketTrajectoryChart` (расширить до двух серий). Никаких «трёх лет вперёд»; старт всегда сегодня.
- **Список** (по строке на хотелку/копилку): тумблер; сумма, дата, ползунок растяжки (взнос/мес ↔ месяцы — одно крутится, второе производное); правка → пересчёт с debounce ~200 мс. Отрисовка: OPEN — пунктир/приглушённо, FIXED — сплошная со значком «в плане». Тумблер FIXED работает в обратную сторону (примерка отказа). Кнопки строки: «зафиксировать», «отложить» (DISMISSED).
- **Ad-hoc строка** «+ а если трата…» (сумма + дата): живёт только в окне; кнопка «сохранить как хотелку» создаёт OPEN-событие.
- **Кнопка «что с капиталом»:** раскрывает месячную картинку капитала/долгов текущего включённого набора (реюз компонентов /wishlist и `GET /wishlist/simulation` + `recompute`); подписана «приблизительно, по месяцам».
- **Вход с Dashboard:** кнопка «примерить» на PocketCard → сюда.
- **Персистентность:** включённые тумблеры, правки параметров, ad-hoc и exclude — в localStorage (переживает перезагрузку, Dashboard не трогает, между устройствами не синхронизируется — осознанно).

## 8. Фиксация

- OPEN-хотелка, растяжка 0: → статус FIXED (дата обязательна — уже есть) → попадает в реальную траекторию как сейчас.
- OPEN-хотелка с растяжкой: → конверсия в TargetFund с `targetDate` и стартовым взносом по ползунку — реюз `WishlistConversionService` (путь «хотелка → копилка» уже написан); новая копилка резервируется по §6.
- Ad-hoc фиксируется только через «сохранить как хотелку» (двухшагово — осознанно, фиксация мимолётной цифры не нужна).
- После фиксации элемент в окне перерисовывается как FIXED, его тумблер переходит в режим «примерка отказа»; localStorage-набор чистится от него.

## 9. Ошибки и валидация

- Хотелка/ad-hoc без даты не включается: фронт запрашивает дату до включения; бэк на `tryOn` без `date` → 400.
- `stretchMonths` вне диапазона §5, `amount ≤ 0`, неизвестный `ref`, невалидный `scope` → 400 (`GlobalExceptionHandler`).
- `exclude` с не-FIXED или конвертированным ref → 400 (защита от рассинхрона окна).
- Пустая примерка валидна: baseline = fitted.
- Failure раскладки одного item не роняет ответ целиком не нужен — вход валидируется заранее; движок тотален.

## 10. Тесты

- **Табличные (раскладка §5):** день первого дохода; месяц без дохода → 1-е; взнос в месяц цели не позже даты; растяжка разовой; клампы диапазона; копейки сходятся; кредитный PMT против существующей формулы.
- **Движок:** новая строка SAVINGS_CONTRIBUTIONS в breakdown-инварианте.
- **Контрактные IT:** `POST /pocket/sandbox` 200 + структура; «включил копилку → fitted.pocket < baseline.pocket»; exclude FIXED-копилки → fitted.pocket > baseline.pocket; **дельта-инвариант §4 день-в-день**; пустая примерка → baseline = fitted; резервирование §6 меняет `GET /pocket`.
- **Vitest:** сложение ползунка (взнос ↔ месяцы), localStorage-набор, «до реализации» появляется/исчезает.
- Смок на реальной базе: Горнолыжка на скоупах «до дохода» (удар 0), «3 мес», «до реализации».

## 11. Вне скоупа (v1)

- Правка существующих планов и «доход −30%» (решение 2026-07-18: не нужно сегменту со стабильной зп).
- Гибридное клиентское сложение линии (контракт готов; отдельная фронт-задача при появлении реальных юзеров/латентности).
- Batch-режим sandbox; ИИ-оптимизатор маршрутов — **ANO-27**.
- Гигиена baseline (просрочка старше якоря задваивается, на данных Кирилла −50 400 из −64 400) — **ANO-28**; примерка от неё не зависит механически, но ценность показа страдает, чинить стоит рядом.
- Синхронизация примерочного набора между устройствами; сохранённые именованные сценарии.
