# Selfin — пересборка продукта (стратегический разбор)

Дата: 2026-06-19. Статус: живой стратегический документ (не спека реализации).

> **Ревизия 2026-06-20.** Ценность из §1 («переживу ли / нулевой порог») пользователь отверг как generic. Контекст: автор приостановил новые фичи, пересматривает идею/бизнес-модель/функционал свежим взглядом. Пользуется на практике 2 функциями из 8 страниц. Цель — связать всё воедино и убрать ощущение «Франкенштейна». Дальше работа ведётся через Linear (эпики → задачи) и Slack.

## 1. Продукт в одной фразе

> **Selfin — форвардный личный финдиректор для российской реальности. Прост на входе для тревожного человека (аванс+зп, рубль, ввод руками или CSV из банка, который уже категоризировал), но с глубиной настоящего CFO — прогноз, сценарии «что если», влияние на поток/капитал/обязательства — доступной по запросу. С ИИ, который реально упрощает и советует, а не висит для галочки.**

Ниша: тревожный mid-month, режим аванс+зарплата (2 дохода в месяц). Главный вопрос — про **будущее** («что меня ждёт»), не про прошлое.

**Ценность и дифференциатор (грунтовано ресерчем 2026-06-21):** форвардное планирование как ценность ВАЛИДИРОВАНО рынком (PocketSmith, Quicken Simplifi, ProjectionLab, PocketGuard), но новизна не наша. Дифференциатор: (a) сегмент РФ — инкумбенты на US bank-aggregation, не покрывают рубль/аванс+зп/ручной+CSV (структурный ров); (b) просто-спереди, глубоко-по-запросу (клин против PocketSmith «мощный-но-сложный» и против РФ-апп «просто-но-прошлое»); (c) ИИ полезный/умный (упрощать ввод + голос CFO на форвард-плане). Открытый угол = forward + просто в РФ. Подтверждено двумя независимыми ресерчами (web + AI), но это сильная гипотеза, не факт — проверять на живых юзерах.

Документальное подтверждение: `docs/specs/01-business-requirements.md` §7 «Two-Salary Horizon» — три точки прогноза («хватит до зп / с чем вхожу в период / выдержу следующий период»). Это исходное сердце продукта.

## 2. Диагноз

- **«Франкенштейн» = части без скелета.** 8 страниц-ровесников, у каждой своя неявная модель мира. Нет общей доменной модели → нет ощущения цельного продукта. Лоскутность не в дизайне, а в отсутствии общего хребта.
- **Треснувший позвонок core loop.** Ценность держится на актуальном журнале; наполнение журнала — мучительный ручной ввод по одной транзакции → пользователь откладывает (раз в 1-2 недели) → журнал устаревает → прогноз врёт → ценность испаряется. Дисциплина страдает из-за трения на входе, а не из-за лени.
- **Несоответствие боли и ценности по разрешению.** Весь продукт затачивает высокоточный учёт прошлого (категории, аккуратность), а ценность — грубоватый прогноз будущего. Пользователь платит высокоточный налог ради прогноза, которому хватило бы агрегатов.

## 3. Core loop (скелет)

> Уточнение 2026-06-21: стержень — не «запись фактов», а **«остаток + план → свободные деньги (кармашек)»**. Журнал (запись фактов) = опциональный слой точности/анализа, его можно вообще не вести. См. §11.

1. **Триггер** — пришла зп (≈2 раза/мес) или прошла неделя-две; тревога «что дальше».
2. **Действие** — открыл, внёс факт (траты/доходы) в журнал.
3. **Выход** — посмотрел «что меня ждёт» (остаток, кассовый календарь, two-salary horizon), подстроил поведение. Иногда подправил план.

Остальные страницы (Analytics, Capital, Strategy, Wishlist, Funds) — **виды на ту же модель**, а не отдельные продукты.

## 4. Доменная модель (черновик)

```
ИСТОЧНИКИ/СЧЕТА → [ОДНА воронка ввода: ручной | импорт] → ЖУРНАЛ → две головы → виды
```

- **Источники/счета** — скрытый слой, которого в коде ещё нет. Дебет, кредит, вклад, наличные. Прогнозу от них нужно узкое: ликвидность + обязательства, не «какая карта что купила».
- **Ввод — ОДНА воронка.** «+» открывает обычный ручной ввод; в том же окне — кнопка массового импорта. Не две параллельные дороги. Дедуп решается на ревью импорта (новое vs уже в журнале).
- **Журнал** — единый источник правды (дата, сумма, категория, источник, флаг долга). Уже существует (`FinancialEvent`).
- **Две головы:** «Анализ — куда ушло» (ПРОШЛОЕ, нужна высокая точность) и «Прогноз — переживу ли» (БУДУЩЕЕ, точность только для крупного/регулярного, хвост мелочи — агрегат).

### 4.1 Связующая ткань: поведенческая модель счёта
Каждый Источник несёт **поведенческую модель**, выбираемую юзером, с разумным дефолтом. Один механизм на все счета → связность без зоопарка частных случаев.
- Дебет = ликвидный (дефолт).
- Кредит = источник долга; можно «неактивный»; перевод с него → авто (долг + приход) в журнал; кормит обязательства/прогноз.
- Вклад = полу-ликвид; НЕ бывает неактивным; входит в капитал; «распечатывается» в прогнозе в трудный момент.
- Наличные = «расход при снятии»; можно «неактивный»; у других иначе → настраивается.

Аналог (проверка по рынку): **YNAB** разделяет *budget accounts* и *tracking accounts*, и отдельно обрабатывает кредитные карты — это ровно «активный/неактивный источник» и «кредит-как-долг». Инстинкт пользователя совпал с проверенным паттерном.

### 4.2 Точность имеет адрес
Точность окупается на **немногом крупном и структурном** (аренда, зп, ЖКХ, платёж по кредиту, вклад) — это «крупные камни», они определяют ответ «переживу ли». На **многом мелком переменном** (продукты, кофе, такси) хватает агрегата. Объединение транзакций — инструмент пересыпать «песок» в один мешок, сохранив «камни». (Финансовый аналог: fixed vs variable; «big rocks»; в инвест-учёте — материальность.)

## 5. UX-закон: прогрессивное раскрытие (вход И выход)

> Всё связано и связь видна, НО в начале настраивать нечего. Стержень бьёт в лицо первым; углубление — по желанию.

**Лестница входа:**
- **День 1 (0 настроек):** ввёл остаток одного счёта (уже есть `BalanceCheckpoint`) + пара трат руками → журнал и дашборд живые, виден объём денег. **Ценность здесь.**
- **Когда захочет:** категории → осмысленный анализ.
- **Продвинутый слой (большинство не доходит):** счета, поведенческие модели, **импорт**.

**Важная поправка (2026-06-19):** «просто кидаешь выписку» — это НЕ день 1. Импорт требует: настроить категории, сопоставить маппинг, понять откуда брать выписку, разобрать чужие счета в выписке. Это сильно потом, продвинутый слой.

**Лестница выхода (симметрично, к разбору позже):** должен быть ОДИН основной вывод-ответ («переживёшь / узко тогда-то»), в который можно изредка углубиться (стратегия, капитал, разбивка). Кернел основного вывода уже есть — two-salary horizon + кассовый календарь.

Принцип: progressive disclosure + smart defaults. Аналоги: Apple HIG / Кен Косиенда «Creative Selection» / Норман «Дизайн привычных вещей»; Круг «Don't Make Me Think». Согласуется с уже записанной [onboarding-стратегией](../../../).

## 8. Ландшафт (анализ «по уму», не в вакууме)

- **YNAB** — zero-based / «give every dollar a job», envelopes, budget vs tracking accounts, спец-обработка кредиток. Силён в дисциплине, слаб в forecast «между зарплатами».
- **Monarch / Copilot Money** — агрегация банков (US) + прогноз/тренды. Авто-ввод снимает «налог», но привязка к банкам и не про тревогу mid-month.
- **Lunch Money / Actual Budget** — импорт CSV + правила, manual+import сосуществуют (наш кейс), инди-аудитория.
- **RU-контекст (Тинькофф/Дзен-мани/CoinKeeper)** — банк уже категоризирует выписку (наш инсайт: данные на столе, продукт заставляет вбивать заново).
- **Финансовые понятия, которые стоит держать в голове:** cash-basis vs accrual (кредитный манёвр), cash-flow forecasting / runway (ядро ценности), zero-based & envelope budgeting, sinking funds (= наши «копилки»/Funds), уровни ликвидности (ликвид/полу-ликвид/неликвид → поведенческие модели счетов).

**Дифференциатор Selfin:** не «учёт» и не «агрегатор», а **прогноз выживания между двумя зарплатами** для тревожного пользователя. Этого толком нет ни у кого — у всех либо бюджет-конверты, либо тренды прошлого.

## 9. Приоритеты и открытые вопросы

Приоритеты:
- Снижение «налога на вход» (импорт как продвинутый слой) — высокий стратегический приоритет, но НЕ день 1.
- Агрегирование транзакций — низкий (нужно лишь в 1 из 3 итогов), отложить.

Открытые узлы (НЕ решать наскоком):
1. **Модель долга/кредитки** — «нужно отточить». Вероятно — выбор поведенческой модели пользователем.
2. **Дедуп** ручной+импорт — на ревью импорта.
3. **Где настраивать категории для импорта** — развилка: в банке (coupling, риск «другие потеряются») vs в selfin (сами ведём онбординг). Гипотеза: в selfin.
4. **Качество ВЫХОДА** — мы разобрали только вход. Что делает прогноз «ответом», а не «данными» — следующий трек.

## 11. Задача (в Linear): кармашек / свободные деньги — re-vision логики

Статус: к заведению в Linear под ANO-5 (Linear MCP временно отключён 2026-06-21).

**Контекст.** Кармашек УЖЕ реализован (`docs/specs/2026-03-22-free-money-calculation.md`, сводный буфер из business-requirements §5, `BalanceCheckpoint`). Задача — НЕ писать с нуля, а пересобрать с правильным видением. Боль пользователя: «каждый раз тыща вопросов, как он считает, и ноль понимания, почему столько» → цель задачи = **прозрачная, объяснимая логика** (рядом с числом — «почему столько» + ближайшая угроза), а не только корректное число.

**Рефрейм стержня (важно).** Кармашек = ОСТАТОК (1 обязательный ввод, `BalanceCheckpoint`) + ПЛАН (форвардные доходы/расходы) → свободные деньги. Историю транзакций можно обнулить / не вести; журнал = опциональный слой точности и анализа, не ядро. Дыра: план-онли дрейфует от реальности → нужен дешёвый ре-якорь остатка (переввести 1 число) либо запись фактов. Прогрессивно: день 1 = остаток + немного плана → кармашек; глубина = запись фактов для точности/анализа.

**Определяющие вопросы для анализа в задаче:**
- Горизонт: дефолт = до следующего дохода (доход вариативен — уточнять точнее); растягивание «+мес / вперёд» — обязательно.
- Что вычитается из «свободных»: плановые траты — да; обязательства и взносы в копилки — это и есть плановые траты (отдельной сущностью можно не делать); неподтверждённые, но намеченные хотелки — показывать отдельной строкой, не съедать кармашек; буфер безопасности — идея есть, реализация TBD.
- Объяснимость: разбивка «почему столько» + ближайшая угроза рядом с числом.
- Визуал: кармашек + календарь-близнец (тот же ответ, нарисованный); продумать отдельно и тщательно.
