# Кармашек: календарь-близнец (ANO-14)

**Дата:** 2026-07-11. **Linear:** ANO-14 (родитель ANO-8), встроенный кусок ANO-23.
**Решение по визуалу:** вариант «график-траектория» (утверждён Кириллом 2026-07-11 по макетам).

## 1. Суть

Кассовый календарь — не отдельная фича, а **тот же ответ кармашка, нарисованный по дням**. ANO-12 уже сделал данные и подключение (Dashboard рендерит `pocket.trajectory`); ANO-14 — это (а) полноценный визуал вместо нищей ленты, (б) возвращение «до второй зп» как скоупа кармашка, (в) смерть последних параллельных расчётов в DashboardService.

Цель визуала (product-rethink §11): «визуальный провал ВИДЕН, а не только читается» — страховка от ложного успокоения одним числом.

## 2. Что уже есть (не трогаем)

- `GET /api/v1/pocket?scope=` → `PocketResultDto`: `pocket`, `currentBalance`, `buffer`, `horizon {type,endDate,label,fallback}`, `minPoint {date,balance,drivenBy}`, `breakdown[]`, `trajectory[{date,balance,income,expense}]`, `wishlistCandidates[]`. Инвариант: `income − expense` = дельта баланса дня; день 0: `expense` = резерв просрочки + плановые траты сегодня, `income = 0`.
- Фронт: `PocketCard` (фетчер `/pocket` для пользовательского скоупа — после §5 Dashboard добавит второй, сторожевой запрос; скоуп-переключатель внутри, отдаёт `PocketResponse` наверх через `onData`) смонтирован на Dashboard и Funds; Dashboard рендерит траекторию в `CashFlowSection` (лента ячеек); алерт разрыва — пока из `fetchDashboard().cashGapAlert`.
- Правило: календарь потребляет ТОЛЬКО ответ `/pocket`. Никаких параллельных расчётов (болезнь ANO-6).

## 3. Визуал: PocketTrajectoryChart

Новый компонент `frontend/src/components/pocket/PocketTrajectoryChart.tsx` — рукописный SVG (прецеденты: ForecastSparkline, графики Strategy; чартовых библиотек не заводим). Чистое представление: props = `PocketResponse` (+ колбэк выбора дня), внутри не фетчит. На Dashboard заменяет текущую ленту `CashFlowSection`; поток данных прежний — лифт из `PocketCard.onData`, рефреш наследуется.

Слои (все из существующих полей ответа):

1. **Линия + заливка остатка** по `trajectory`. Точки-маркеры видны на коротких горизонтах, на длинных сливаются в линию.
2. **Ноль всегда включён в масштаб Y** (домен `[min(0, minBalance), max(0, maxBalance)]` — ноль остаётся в кадре и когда вся траектория под водой): расстояние до дна — главный сигнал; ниже нуля — красная зона «разрыв».
3. **Буфер**: пунктирная линия + янтарная зона `0..buffer`. Рендерится только при `buffer > 0`.
4. **Минимум**: маркер на `minPoint` + аннотация «мин {dd.MM} · {сумма} · {drivenBy}» (виновник опускается, если `drivenBy == null`).
5. **Доходы**: метки на днях с `income > 0` — восстановление после провала видно глазами.
6. **Сегодня**: маркер первой точки. Ось X — редкие тики: сегодня, минимум, дни доходов, конец горизонта (с разрешением коллизий, без сплошной гребёнки дат).
7. **Тап/ховер по дню**: вертикальная направляющая + строка деталей «{дата} · остаток X · +доход / −расход» из агрегатов точки. Поимённый состав дня в v1 НЕ делаем (в траектории только агрегаты; поимённая детализация — с ANO-16).
8. **Fallback-горизонт** (`horizon.fallback == true`): шапка показывает честный лейбл. Для NEXT_INCOME-fallback маркеров дохода нет по построению (в 92 днях не нашлось ни одного); для SECOND_INCOME-fallback известный первый доход МОЖЕТ попасть в горизонт — и должен быть виден маркером (см. §4).

Шапка блока: «Кассовый календарь · {horizon.label}» — скоуп графика ВСЕГДА равен скоупу числа и фразы (один ответ, нарисованный дважды).

**§3.9 Минимальное окно календаря (дополнение 2026-07-16, ANO-24).** Боль: доход завтра → горизонт «до дохода» = 2 дня → календарь неинформативен (Кирилл, 14.07). Решение — **информационный хвост**: `trajectory` продлевается до `max(horizonEnd, asOf + 7 дней)` (константа `MIN_CALENDAR_DAYS` в PocketEngine), при этом:
- Число/минимум/breakdown НЕ меняются: min-трекинг только внутри `horizonEnd` — семантика «до дохода» нетронута.
- Хвост = только плановые события, БЕЗ размаза прогноза незапланированных (окно размаза остаётся `min(конец месяца, horizonEnd)`: разбавление дневной доли на длинном окне меняло бы само число).
- Фраза «После дохода → X» берёт точку траектории на `horizon.endDate`, а НЕ последнюю (с хвостом это разные дни).
- График рисует хвост приглушённо, с пунктирной границей горизонта и пометкой в легенде. Цветовые токены страницы: `--color-success/-danger/-surface/-border`, янтарный для зоны буфера — в тон существующей тёмной теме.

Размещение: только Dashboard (IA 2026-03-10). Funds остаётся с компактным PocketCard без графика. Пилюля «2-й доход» при этом появится и на Funds (PocketCard общий) — это безвредно: кап перевода в копилку берётся из `trajectory[0].balance`, а день 0 скоуп-инвариантен.

## 4. Скоуп SECOND_INCOME («до 2-го дохода»)

Семантика кармашка, не старого Dashboard: **второй будущий плановый доход ЛЮБОЙ категории** — никакого детекта «Зарплата» по имени (запрет спеки кармашка §4; аванс+зп и так дают две даты).

Бэкенд:
- `PocketScope.Type` += `SECOND_INCOME`; parse строки `"SECOND_INCOME"`; невалидное — прежние 400.
- Репозиторий: запрос **двух ближайших различных дат** плановых доходов в окне `(asOf, asOf+92]` (условия те же, что в `findNextPlannedIncomeDate`: PLAN/PLANNED, INCOME, не удалён, `wishlistStatus IS NULL`; DISTINCT по дате — две суммы в один день = одна дата).
- Резолюция горизонта в `PocketService`: вторая дата найдена → `horizonEnd` = она (включительно, как у NEXT_INCOME — день дохода виден в траектории).
- **Fallback SECOND_INCOME — правдивый, свой:**
  - 0 дат в окне → идентично NEXT_INCOME-fallback: `asOf+30`, `fallback=true`, существующий лейбл «30 дней вперёд (нет плановых доходов)» — он правдив, доходов действительно нет.
  - ровно 1 дата → `horizonEnd = max(датаПервогоДохода, asOf+30)`, `fallback=true`, НОВЫЙ лейбл «до dd.MM (второй доход не найден)». Горизонт всегда накрывает известный доход (его маркер виден на графике — правдиво), лейбл не врёт про «нет плановых доходов». Существующий лейбл для этого случая запрещён (у проекта явная планка правдивых формулировок, ср. фикс a803eae).
- Лейбл в `PocketEngine.horizonLabel` (не-fallback): «до 2-го дохода dd.MM».
- `pocketPhrase`: fallback-ветка сейчас безусловно говорит «плановых доходов нет» — для SECOND_INCOME-fallback использовать `horizon.label` ДОСЛОВНО в обоих подслучаях (оба лейбла правдивы; фронт не должен угадывать подслучай по содержимому label).
- Доставка подслучая в движок: `horizonLabel` живёт в `PocketEngine`, а boolean `horizonFallback` не различает подслучаи → `PocketInput.horizonFallback` заменяется enum'ом `FallbackKind { NONE, NO_INCOMES, SECOND_NOT_FOUND }` (резолвится в `PocketService`); в DTO `Horizon.fallback` остаётся boolean (`kind != NONE`) — фронтовый контракт не меняется.

Фронт:
- `PocketScopeType` += `'SECOND_INCOME'`; пилюля «2-й доход» в `SCOPES` (`PocketCard`). Порядок: До дохода · 2-й доход · 3 мес · 6 мес.
- `pocketPhrase`: горизонт SECOND_INCOME заканчивается днём дохода → хвост «После дохода → X» работает как у NEXT_INCOME; ветвление расширить, тесты дополнить.

## 5. Алерт ← minPoint, смерть зарплатных расчётов DashboardService (кусок ANO-23)

- **Алерт = сторож на фиксированном растянутом скоупе.** Старый `detectCashGap` сканировал до ВТОРОЙ зарплаты независимо от того, куда смотрит пользователь — терять эту защиту нельзя (разрыв между авансом и зп = ядро обещания продукта). Поэтому: Dashboard, помимо пользовательского скоупа PocketCard, делает один дополнительный запрос `GET /pocket?scope=SECOND_INCOME` («сторожевой скоуп») и рендерит алерт из ЕГО `minPoint` при `minPoint.balance < 0` — дата, дефицит `|balance|`, виновник `drivenBy` (если есть). Это НЕ параллельный расчёт (тот же движок, второе представление); рефреш — тем же `refreshSignal`.
- Если дата сторожевого разрыва за концом текущего горизонта графика — алерт дописывает подсказку «переключись на "2-й доход"» (auto-switch скоупа в v1 не делаем, PocketCard остаётся неконтролируемым).
- **Осознанная смена семантики (даты):** старый `detectCashGap` = ПЕРВЫЙ отрицательный день, `minPoint` = САМЫЙ ГЛУБОКИЙ (при равенстве — ранний). При двух провалах даты разойдутся. Принято: одна истина = minPoint (ANO-13-фраза уже так считает); ранний неглубокий провал виден на графике.
- Empty-state условие Dashboard (`Dashboard.tsx:363` сейчас ссылается на `cashGapAlert`) переводится на наличие сторожевого алерта.
- Выпилить из `DashboardService`: детект зарплат по категории «Зарплата», salaryDates, calcBalanceBefore/After, `detectCashGap`, gapHorizon, `FORECAST_HORIZON_DAYS`; из `DashboardDto`: `nextSalaryDate`, `balanceBeforeNextSalary`, `balanceAfterNextSalary`, `secondSalaryDate`, `balanceBeforeSecondSalary`, `cashGapAlert`, `endOfMonthForecast` (фронт ничего из этого не рендерит после ANO-13 — перед удалением подтвердить grep'ом; `currentBalance` оставить, если ещё потребляется). Зеркальная чистка `DashboardData`/`CashGapAlert` в `frontend/src/types/api.ts`.
- После этого дашборд-endpoint отвечает только за progressBars (+currentBalance при необходимости) — последний зарплатный детект по имени категории умирает; любые «до второй зп» смыслы дальше живут только в механизме скоупов кармашка.

## 6. Попутный фикс: recurring на растянутых скоупах

`PocketService` не продлевает бесконечные recurring-правила — на MONTHS:3/6 (и теперь SECOND_INCOME) траектория молча теряет повторяющиеся события за пределами материализованных строк. Хуже: нематериализованный recurring-ДОХОД может уронить NEXT_INCOME/SECOND_INCOME в fallback, хотя правило существует. Фикс: в начале `getPocket`, ДО резолюции горизонта и выборки дат доходов, вызвать `extendIndefiniteRules(asOf + 36 мес)` — зеркально `FundPlannerService:41`, включая non-fatal try/catch (сбой продления не должен ронять read-запрос; `extendIndefiniteRules` — REQUIRES_NEW, из readOnly-транзакции работает). С тестом, доказывающим, что без вызова событие терялось/скоуп падал в fallback, а с вызовом — траектория и горизонт корректны. Перф-заметка: продление теперь срабатывает на каждый `/pocket` (на Dashboard — дважды за загрузку: пользовательский скоуп + сторож); для single-user это дешёвый no-op, оптимизация не нужна.

## 7. Отложено сознательно

- Поле `forecast` в точке траектории (отдельная штриховка прогнозного размаза) — YAGNI v1, breakdown уже объясняет.
- Поимённые события в деталях дня — вместе с ANO-16.
- Схлопывание алерт-карточки в hero — после живого использования.
- Продление окна прогноза незапланированных трат за текущий месяц — ANO-22.

## 8. Тесты

Бэкенд:
- `PocketScopeTest`: parse `SECOND_INCOME`, невалидные строки.
- `PocketServiceTest`: резолюция горизонта — вторая дата есть / только одна (fallback-лейбл «второй доход не найден», горизонт накрывает первую дату) / ни одной (обычный fallback); включительность конца; продление recurring ДО резолюции (доход, материализуемый продлением, якорит горизонт, а не роняет в fallback).
- Repository-IT: запрос двух различных дат (два дохода в один день = одна дата).
- `PocketEngineTest`: лейбл «до 2-го дохода».
- Recurring-фикс: интеграционный тест «правило без конца + горизонт 3 мес → события в траектории».
- `PocketControllerIT`: `?scope=SECOND_INCOME` end-to-end.
- DashboardService: тесты выпиленного удаляются; дельта-ассерт, что endpoint жив и отдаёт progressBars.

Фронт (vitest):
- `pocketPhrase`: SECOND_INCOME-лейбл и хвост «После дохода»; SECOND_INCOME-fallback с одной датой — фраза использует label дословно, без «плановых доходов нет».
- `PocketTrajectoryChart`: домен Y включает ноль (в т.ч. вся траектория под водой); зоны (разрыв/буфер) появляются только когда должны; выбор тиков без коллизий; аннотация минимума с/без drivenBy; строка деталей дня.
- Алерт-сторож: рендер из minPoint сторожевого скоупа при `balance < 0`, отсутствие при `>= 0`; подсказка про переключение, когда дата за горизонтом графика.

## 9. Definition of done

- На Dashboard вместо ленты — график-траектория со слоями из §3, скоуп общий с числом и фразой.
- Скоуп «2-й доход» работает end-to-end (пилюля → бэкенд → лейбл → фраза → график).
- Алерт разрыва питается minPoint сторожевого скоупа SECOND_INCOME (защита «между авансом и зп» сохранена без пользовательских действий); в DashboardService не осталось ни зарплатного детекта, ни собственной траектории.
- Recurring-события не теряются на растянутых горизонтах.
- Все тесты зелёные; ни одного нового расчёта истины вне PocketEngine.
