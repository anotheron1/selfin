# [FEA-001] Начальный баланс счёта (Balance Checkpoint)

**Дата:** 2026-03-08

---

## 0. Проработка идеи

### 0.1 Опиши идею в свободной форме
Нужно ввести в систему актуальный остаток на счёте — ту сумму, которая реально
лежит в банке на конкретную дату. Без этого приложение считает бюджет «с нуля»,
и все цифры — баланс, прогноз, кассовые разрывы — неверны.

### 0.2 Какая боль/неудобство стоит за идеей?
Все расчёты стартуют от нуля, поэтому Dashboard показывает бессмысленные цифры.
Текущий обходной путь — создать фиктивную транзакцию-«доход» на нужную сумму —
это костыль, который засоряет данные.

### 0.3 Что происходит без этой фичи?
- Данные неверные: расчёты считают от нуля.
- Обходной путь: создаю фиктивную транзакцию.
- Приложение в принципе не работает корректно без стартового баланса.

### 0.4 Как часто будет обновляться?
- [x] Регулярно — раз в неделю/месяц (сверка с банком)

### 0.5 На что в системе влияет?
- **Dashboard**: Hero-метрика «Текущий баланс» и «Прогноз на конец месяца»
- **Budget**: расчёт кассового разрыва по дням
- **Analytics**: всё, что считается от остатка

### 0.6 Что видит пользователь после реализации?
В **Settings** — новый раздел «Баланс счёта»:
- Поле ввода суммы + поле даты → кнопка «Сохранить»
- Список предыдущих записей (дата + сумма), каждую можно удалить или отредактировать

### 0.7 Вход / Выход
- **Вход**: дата (date), сумма (amount)
- **Выход**: «точка отсчёта» — система берёт самый свежий чекпоинт и считает
  от него все последующие события

### 0.8 Аналог в других приложениях
Не указан.

### 0.9 Граничные случаи
- **Дата в прошлом**: чекпоинт создан на 1 марта, сегодня 8 марта —
  система должна учитывать все события за эти 7 дней.
- **Несколько чекпоинтов**: берётся самый свежий по дате (не по дате создания).
- **Удаление записи**: если удаляем последний чекпоинт — баланс возвращается
  к следующему по дате (или к нулю).
- **Редактирование**: изменение суммы / даты пересчитывает всё downstream.
- **Отрицательный баланс**: допускается (кассовый разрыв это подтверждает).

### 0.10 MVP фичи
Поле суммы + дата + история изменений (просмотр + редактирование + удаление).
Ничего лишнего.

## 1. Цель фичи

Позволить пользователю зафиксировать реальный остаток на счёте на конкретную
дату, чтобы все расчёты кассового календаря (баланс, прогноз, кассовые разрывы)
стартовали от этой точки, а не от нуля.

---

## 2. Пользовательский сценарий

**Добавление:**
Как пользователь, я хочу ввести сумму и дату текущего остатка в Настройках,
чтобы Dashboard показывал корректные цифры.

Шаги:
1. Открыть Settings → раздел «Баланс счёта»
2. Ввести сумму (например, 87 500 ₽) и дату (например, 2026-03-08)
3. Нажать «Сохранить»
4. Dashboard обновляется: Hero-метрика теперь считается от этой суммы

**Обновление (сверка с банком):**

1. Открыть Settings → «Баланс счёта»
2. Добавить новую запись с сегодняшней датой и текущей суммой из банка
3. Расчёты пересчитываются автоматически

---

## 3. Критерии приёмки (Acceptance Criteria)

- [ ] В Settings есть раздел «Баланс счёта» с формой: поле суммы + поле даты
- [ ] После сохранения запись появляется в истории
- [ ] Dashboard: «Текущий баланс» = чекпоинт + Σ(факт-события до сегодня; где факта нет — план)
- [ ] Dashboard: «Прогноз на конец месяца» = текущий баланс + Σ(план-события от завтра до конца месяца)
- [ ] Если чекпоинтов нет — баланс = 0 (поведение как сейчас)
- [ ] Берётся чекпоинт с максимальной датой (не с максимальным `created_at`)
- [ ] Запись в истории можно удалить; после удаления перерасчёт
- [ ] Запись в истории можно отредактировать (сумма и/или дата); после — перерасчёт
- [ ] Отрицательные значения суммы не допускаются (UI блокирует)

## 4. Out of scope (MVP)

- Автоматический импорт из банка
- Несколько счетов (один глобальный чекпоинт)
- Уведомления «пора обновить баланс»

## 5. Технический scope

### Формулы расчёта (решено)

Dashboard показывает **две отдельные метрики** с разной логикой:

**① Текущий баланс (факт)**
```
factBalance =
  latestCheckpoint.amount
  + SUM(fact_amount  WHERE date >= latestCheckpoint.date AND date <= today AND fact_amount IS NOT NULL)
  + SUM(planned_amount WHERE date >= latestCheckpoint.date AND date <= today AND fact_amount IS NULL)
```
> Для дат до сегодня: если факт есть — берём факт, иначе берём план.
> Результат — «где мы сейчас, с учётом всего прошедшего».

**② Плановый баланс на конец месяца (прогноз)**
```
plannedBalance =
  factBalance
  + SUM(planned_amount WHERE date > today AND date <= endOfMonth)
```
> Берём текущий факт-баланс и добавляем все запланированные события до конца
> месяца. Результат — «к чему придём, если план выполнится».

Оба значения уже отображаются на Dashboard («Текущий баланс» и «Прогноз на конец
месяца»). После фичи они просто начнут считаться от реальной точки, а не от нуля.

```sql

  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  date       DATE         NOT NULL,
  amount     NUMERIC(19,2) NOT NULL CHECK (amount >= 0),
  created_at TIMESTAMPTZ  NOT NULL DEFAULT now(),

);
```
