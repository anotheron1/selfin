# Модель данных (MVP)

Ядро системы Личного платёжного календаря строится на следующих реляционных сущностях:

## 1. Категория (Category)
Справочник статей бюджета (доходов и расходов).
- `id` (UUID/Long) - PK
- `name` (String) - Название (например, "Коммуналка", "Садик")
- `type` (Enum: INCOME, EXPENSE)
- `is_mandatory` (Boolean) - Пометка обязательности по умолчанию

## 2. Финансовое событие (FinancialEvent)
Основная операционная единица.
- `id` (UUID/Long) - PK
- `date` (LocalDate) - Дата (плановая или фактическая)
- `category_id` (FK -> Category)
- `type` (Enum: INCOME, EXPENSE, FUND_TRANSFER)
- `planned_amount` (BigDecimal) - Плановая сумма
- `fact_amount` (BigDecimal) - Фактическая сумма (может быть `null` для еще не наступивших событий)
- `status` (Enum: PLANNED, EXECUTED, CANCELLED)
- `is_mandatory` (Boolean) - Признак обязательности (копируется из Категории, но может быть переопределен)
- `description` (String) - Комментарии и пояснения пользователя

## 3. Целевой фонд (TargetFund)
Сущность для накопления средств.
- `id` (UUID/Long) - PK
- `name` (String) - Имя фонда ("Отпуск", "Резерв", "Кармашек")
- `target_amount` (BigDecimal) - Желаемая сумма пополнения. Для "Кармашка" может быть не ограничена (null).
- `current_balance` (BigDecimal) - Текущий накопленный итог
- `status` (Enum: FUNDING, REACHED)
- `priority` (Integer) - Порядок, в котором фонды забирают профицит

## 4. Операция изменения фонда (FundTransaction)
Связующая таблица для аудита перемещений "Бюджет -> Фонд".
- `id` (UUID/Long) - PK
- `fund_id` (FK -> TargetFund)
- `source_event_id` (FK -> FinancialEvent) - Какое событие выделило средства (например, сэкономили на фильтре 2100р - создали транзакцию в фонд на +2100р)
- `amount` (BigDecimal)
- `transaction_date` (LocalDate)

## 5. Период (Периодичность)
Отчетные периоды (Год, Месяц, Неделя) в базе данных могут не храниться жестко, а вычисляться "на лету" из дат событий (с помощью `GROUP BY date_trunc`). Возможна реализация таблицы `BudgetPeriod` для фиксации требуемых плановых доходов на конкретный месяц и хранения "целей/таргетов".

## 6. Чекпоинт баланса (BalanceCheckpoint) — FEA-001

Точка отсчёта для расчёта текущего баланса и прогноза. Пользователь периодически
(раз в неделю/месяц) синхронизирует остаток с реальным банковским счётом.

- `id` (UUID) - PK
- `date` (LocalDate) - Дата, на которую зафиксирован баланс
- `amount` (BigDecimal ≥ 0) - Реальный остаток на счёте
- `created_at` (LocalDateTime)
- `updated_at` (LocalDateTime)

**Логика использования:**
Берётся запись с максимальным `date` (не `created_at`). Это значение служит
стартом для расчёта:

```
Текущий баланс = amount + Σ(события от date до сегодня)
Прогноз        = Текущий баланс + Σ(плановые события от завтра до конца месяца)
```

Если чекпоинтов нет — баланс считается от нуля.
