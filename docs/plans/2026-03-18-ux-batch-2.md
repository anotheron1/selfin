# UX Improvements Batch 2

**Дата:** 2026-03-18

Документ описывает набор доработок, реализованных в рамках второй очереди UX-улучшений. Изменения разделены на бэкенд (B) и фронтенд (F).

---

## B1: Наследование приоритета из категории

**Что изменилось:** Новые события теперь наследуют приоритет категории, если приоритет явно не задан. Flyway-миграция проставила правильный приоритет существующим событиям.

**Зачем:** Блок "Обязательные траты" на странице Analytics фильтрует события по приоритету HIGH. До исправления события создавались с MEDIUM по умолчанию, даже если категория имела HIGH — блок показывал неверные данные.

**Как работает:** При создании события, если поле `priority` не передано, сервис берёт `priority` из связанной категории. Если категория тоже без приоритета — используется MEDIUM.

---

## B2: factExpenses в ответе планировщика фондов

**Что изменилось:** Endpoint `GET /api/v1/funds/planner` теперь возвращает поле `factExpenses` (nullable) для текущего месяца.

**Зачем:** Фронтенд использует это значение для отображения точки фактических расходов на графике сбережений (см. F7).

**Как работает:** Если запрашиваемый месяц совпадает с текущим, сервис суммирует фактические расходы по всем событиям с `factAmount > 0` и возвращает их в поле `factExpenses`. Для будущих месяцев значение `null`.

---

## F1: Страница "Журнал" (была "Бюджет")

**Что изменилось:**
- Страница переименована из "Бюджет" в "Журнал".
- Убран блок сводных итогов вверху страницы.
- События с одинаковым названием внутри месяца объединяются в одну карточку.

**Зачем:** Страница выполняет роль журнала событий, а не инструмента планирования. Блок итогов был избыточен для этой цели. Слияние дублирующихся карточек убирает визуальный шум от повторяющихся операций.

**Как работает:** При рендере список событий за месяц группируется по полю `description` (непустое). Если в группе ровно два события — одно с `factAmount`, другое без — они сливаются в одну карточку, показывающую `plannedAmount`, `factAmount` и дельту. Позиция карточки определяется по более ранней дате.

## F2: Полное редактирование события

**Что изменилось:** `EditEventSheet` теперь позволяет редактировать все поля события: название, плановая сумма, фактическая сумма, дата, категория, тип, приоритет.

**Зачем:** Раньше в шторке редактировались только фактическая сумма и комментарий. Пользователь не мог исправить ошибку в названии или изменить дату без удаления события.

**Как работает:** Форма в `EditEventSheet` расширена до полного набора полей. При сохранении вызывается `PUT /api/v1/events/{id}` с обновлёнными данными.

---

## F3: Редактирование позиций вишлиста

**Что изменилось:** У каждой позиции вишлиста появилась кнопка-карандаш, открывающая форму редактирования: название, сумма, URL, дата.

**Зачем:** Позиции вишлиста не редактировались после создания — только удалялись и создавались заново.

**Как работает:** Кнопка открывает модальное окно с заполненными текущими значениями. При сохранении вызывается `PUT /api/v1/wishlist/{id}`. Поле URL отображается в форме, но бэкенд пока его не сохраняет — ограничение зафиксировано.

---

## F4: Семантические состояния план/факт на Dashboard

**Что изменилось:** Вместо цветовой шкалы по проценту выполнения (зелёный / жёлтый / красный) введены пять именованных состояний:

| Состояние | Условие | Цвет |
|---|---|---|
| Выполнено | факт 95–105% от плана | зелёный |
| В процессе | 0 < факт < 95% плана | синий |
| Не выполнено | план > 0, факт = 0 | серый |
| Не запланировано | план = 0, факт > 0 | янтарный |
| Перерасход | факт > 105% плана | красный |

**Зачем:** Процентная шкала не различала ситуации "деньги ещё не потрачены" и "деньги не потрачены намеренно". Семантические состояния дают пользователю однозначное прочтение статуса.

---

## F5: Подзаголовок блока "Обязательные траты" в Analytics

**Что изменилось:** Под заголовком блока добавлен подзаголовок с пояснением, что именно показывает блок.

**Зачем:** Пользователи не понимали, по какому критерию события попадают в этот блок (фильтр по приоритету HIGH).

---

## F6: Расчёт лимита распределения

**Что изменилось:**
- Потолок распределения изменён с фиксированных 50% дохода на остаток после всех плановых расходов.
- Блок показывает три строки: "Плановый доход", "Остаток (обяз.)" (после вычета обязательных расходов), "Остаток (все расходы)".
- Режим "сверхурочных" удалён.

**Зачем:** Фиксированные 50% не учитывали реальный бюджет пользователя. Новый расчёт показывает, сколько средств доступно для распределения по фондам после покрытия запланированных расходов.

**Как работает:** Фронтенд вычитает суммарные плановые расходы из планового дохода и передаёт полученное значение как доступный лимит.

---

## F7: Точка фактических расходов на графике сбережений

**Что изменилось:** На графике сберегательного фонда в позиции текущего месяца появляется красная точка, отображающая фактические расходы.

**Зачем:** График показывал только прогнозные значения. Точка факта позволяет сравнить план и реальность прямо на графике.

**Как работает:** Компонент читает `factExpenses` из ответа `GET /api/v1/funds/planner` (см. B2). Если значение не `null`, точка рисуется на оси Y, соответствующей сумме расходов.

---

## F8: Привязка кредитных слайдеров к PMT

**Что изменилось:** Слайдеры кредитного фонда добавляют специальную точку привязки: 0 → PMT (ежемесячный обязательный платёж) → обычные шаги. При остановке на значении PMT показывается метка "Ежемесячный платёж: X ₽ (Y%)".

**Зачем:** Пользователям удобно выставить взнос ровно в размере обязательного платежа. Без привязки это было затруднено из-за дискретности шагов слайдера.

**Как работает:** Логика слайдера вставляет значение PMT как дополнительный шаг между 0 и первым обычным значением. При совпадении текущего значения с PMT отображается пояснительная метка.

**Снимки бюджета:** оставлены без изменений. Интерфейс для сравнения исходного плана с фактом будет реализован в отдельной итерации.
